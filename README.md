## 🎯 프로젝트 개요

### 목표
Indy7 로봇 팔에 ㄴ자형 타격 도구를 장착하고, **자율 상태 머신(SCAN → THINK → STRIKE → OBSERVE)**을 통해 3쿠션 당구를 수행합니다.

- 수구(흰공)로 황구·적구 모두 접촉
- 총 쿠션 반사 2회 이상 (2쿠션/3쿠션 모두 유효)
- 장애물 없이 3회 시도 → **3/3 성공률** 달성

### 핵심 기술
| 기술 | 설명 |
|------|------|
| **PyBullet 공 전용 시뮬레이션** | 720각도 × 3속도 = 2160개 후보를 헤드리스 PyBullet에서 직접 탐색 |
| **각도 우선순위 정렬** | 실측 기반 안전 각도(50-180°) 우선 선택으로 글랜싱 블로우 제거 |
| **2쿠션 우선 전략** | 궤적이 짧은 2쿠션을 우선 → 예측 정확도 향상 (필요 시 3쿠션 자동 전환) |
| **벽 뚫림 방지** | safe_approach_dist 동적 계산 → 도구가 벽을 뚫지 않는 안전 궤적 |
| **Damped Least Squares IK** | 5회 반복 수렴 + 관절한계/특이점/바디뚫림 검증 |
| **PD 토크 제어** | Kp=5000, Kd=200 부스트로 1.0 m/s 수평 타격 달성 |
| **ㄴ자 도구 수평 타격** | TOOL_VERTICAL_DROP=6cm + TOOL_HORIZONTAL_EXT=3cm 오프셋 자동 계산 |
| **자동 캘리브레이션** | 좌표 보정(Phase 1) + 물리 파라미터 최적화(Phase 2) 완전 자동 |

## 🏗️ 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                  State Machine (FSM)                    │
│  ┌──────┐   ┌───────┐   ┌────────┐   ┌─────────┐      │
│  │ SCAN │──▶│ THINK │──▶│ STRIKE │──▶│ OBSERVE │──┐   │
│  └──────┘   └───────┘   └────────┘   └─────────┘  │   │
│      ▲                                      │      │   │
│      └──────── (retry if miss) ─────────────┘      │   │
│      └──────── (next round if success) ────────────┘   │
└────────────┬───────────────┬──────────────┬────────────┘
             │               │              │
     ┌───────▼───────┐ ┌────▼────┐  ┌──────▼──────┐
     │ CushionPlanner│ │Traj.Plan│  │ RobotControl│
     │ (공 시뮬)     │ │(SE3 궤적)│  │ (PD 토크)   │
     └───────────────┘ └─────────┘  └─────────────┘
             │                              │
     ┌───────▼───────┐              ┌──────▼──────┐
     │ PyBullet      │              │ Pinocchio   │
     │ (Headless)    │              │ (FK, Jac)   │
     └───────────────┘              └─────────────┘
```

### 작동 흐름

1. **SCAN**: PyBullet에서 3공 위치 + 테이블 bounds 읽기 (실환경: 카메라 + detect_balls)
2. **THINK**: 720각도 × 3속도 공 전용 시뮬레이션 → 안전 각도 + 2쿠션 우선 정렬 → 최대 25개 후보
3. **STRIKE**: 후보별 IK 검증 → 첫 유효 후보 실행 → 즉시 Home 복귀
4. **OBSERVE**: 접촉 이벤트 추적 → 총 쿠션 ≥ 2 + 양 타겟 히트 → 성공

## 🔍 핵심 모듈

### 1. `cushion_planner.py` — 공 전용 시뮬 기반 탐색

```
1. Grid Search: 720각도(0.5°) × 3속도 = 2160개 후보
2. PyBullet 헤드리스에서 3공 직접 시뮬레이션
3. 각도 우선순위 정렬 (글랜싱 블로우 방지):
   - 50-180°: 안전 (우선순위 0) ← 항상 정확한 타격
   - 30-50°, 180-330°: 보통 (우선순위 1)
   - 0-30°, 330-360°: 위험 (우선순위 2) ← 글랜싱 블로우 빈발
4. 같은 우선순위 내에서 쿠션 수 적은 것 우선 (2쿠션 > 3쿠션)
5. 벽 뚫림 방지: safe_approach_dist 동적 계산
6. 5° 각도 다양성 필터 → 최대 25개 후보
```

### 2. `state_machine.py` — 자율 상태 머신

**성공 기준**: 양 타겟 히트 + 전체 이벤트에서 총 쿠션 ≥ 2 (2쿠션/3쿠션 모두 유효)

**후보 선택**: IK 검증 통과하는 첫 후보 실행
- Manipulability ≥ 0.015 (글랜싱 블로우 방지)
- 바디 뚫림 z < 0.30m
- 관절 점프 < 0.3 rad
- 후보별 safe_approach_dist 적용 (벽 안전)

### 3. `trajectory_planner.py` — ㄴ자 도구 수평 타격

```
ㄴ자 도구 형상:
  EE
   |  (6cm 수직)
   |
   └──● (3cm 수평, 큐팁 직경 13mm)

궤적: Home → 상공(25cm) → Ready(공 뒤 8~10cm) → Impact(공) → Follow(5cm, bounds 클램프) → Home
```

### 4. `calibration_loop.py` — 완전 자동 캘리브레이션

```
Phase 1 (좌표): 카메라 검출 → 로봇 접근 → 접촉 여부로 오프셋 자동 보정
Phase 2 (물리): 자동 타격 → 카메라 관측 → Nelder-Mead로 6개 파라미터 최적화
               (반발계수, 마찰, 속도전달비, 테이블 위치 보정)
```

### 5. `ik_solver.py` — 역기구학 솔버

- 5회 반복 DLS IK 수렴 (1회만 하면 곡선 궤적 발생)
- 궤적 전체에서 10% 샘플링하여 검증
- 관절한계, 특이점, 바디뚫림, 관절점프 4단계 검사

## 🎱 왜 이 프로젝트는 어려운가

> **"위치 알면 공식대로 치면 되는 거 아닌가요?"**
>
> 이 프로젝트는 Pick-and-Place가 아닙니다.

### 근본적 차이: 정적 배치 vs 동적 충격

| 비교 항목 | Pick & Place | 당구 타격 |
|-----------|-------------|-----------|
| **제어 대상** | 위치 (x, y, z) | 위치 + 속도 + 방향 + 타이밍 |
| **동작 속도** | 느림 (~0.1 m/s) | 빠름 (**1.0+ m/s**) |
| **접촉 성격** | 정적 파지 (grasp) | **충격 (impulse, <1ms)** |
| **오차 전파** | 없음 (놓으면 끝) | **기하급수적 증폭** |
| **물리 모델** | 불필요 | 반발계수, 마찰, 스핀 전부 필요 |
| **계획 복잡도** | Point-to-point | **조합 탐색 (720각도 × 물리 시뮬)** |
| **되돌릴 수 있나?** | 즉시 재배치 | 불가능 (공은 이미 굴러감) |

Pick-and-place에서 1mm 오차는 그대로 1mm입니다.
당구에서 **0.5° 각도 오차**는 2쿠션 후 **수십 cm**의 궤적 차이로 증폭됩니다.

### "공식대로 하면 된다"가 틀린 5가지 이유

#### 1️⃣ 공식이 가정하는 이상적 조건은 현실에 없다

교과서 쿠션 반사 공식: **입사각 = 반사각**

현실:
- 쿠션 고무의 탄성은 **비선형** (입사 속도에 따라 반발계수가 변함)
- 공이 쿠션에 **파묻히며** 접촉 시간 동안 회전(스핀)이 변함
- 쿠션 높이가 공 중심보다 약간 높아서 **하향 스핀**이 생김
- 테이블 천 마찰은 **방향에 따라 다름** (결이 있음)
- 같은 테이블이라도 온도·습도에 따라 마찰이 달라짐

> 결론: 공식은 1쿠션까지는 근사적으로 맞지만, 2쿠션 이상에서는 **매번 다른 결과**가 나옵니다. 그래서 물리 시뮬레이터(PyBullet)로 직접 시뮬레이션 + 현실 데이터로 파라미터를 캘리브레이션해야 합니다.

#### 2️⃣ 로봇 팔은 당구 큐가 아니다

프로 당구 선수는 큐를 잡고 **시선 → 공 → 목표**를 일직선으로 맞춘 뒤 부드럽게 밀어칩니다.

로봇 팔은:
- **6개 관절의 회전 조합**으로 직선 운동을 만들어야 함 (역기구학)
- 타격 순간 **특이점(singularity)** 근처에서 관절 속도가 발산할 수 있음
- ㄴ자 도구 끝의 실제 궤적은 EE 궤적과 **3cm × 6cm 오프셋** 관계
- 관절 토크 한계 때문에 원하는 속도를 못 낼 수 있음
- PD 컨트롤러의 추종 오차: 1.0 m/s에서 **1.2~1.7mm** 위치 오차

> Pick-and-place는 속도가 느려서 PD 오차가 무시 가능. 당구 타격은 **충격 직전의 0.1mm가 결과를 바꿉니다**.

#### 3️⃣ "칠 수 있는 각도"가 생각보다 적다

이론적으로 360° 모든 방향으로 칠 수 있지만:

- **벽 근처 공**: 도구가 벽을 뚫을 수 없어 접근 불가 → 30~50%의 각도가 차단됨
- **로봇 작업 공간**: 테이블 좌측 끝(x<0.10m)은 로봇 팔이 닿지 않음
- **특이점 회피**: 팔이 완전히 펴진 자세(singularity)에서는 IK 불가
- **ㄴ자 도구 충돌**: 도구 자체가 테이블에 닿아서 접근 불가한 자세 존재
- **글랜싱 블로우**: 0~30° 범위에서 도구가 공을 빗겨쳐서 속도 전달 실패

실측 결과: 평균적으로 전체 후보 중 **15~25%만** IK+충돌+벽 검사를 통과합니다.

#### 4️⃣ 시뮬레이션과 현실의 괴리 (Sim-to-Real Gap)

시뮬레이터에서 완벽한 3쿠션 궤적을 찾아도:

| 요인 | 시뮬 | 현실 |
|------|------|------|
| 마찰계수 | 고정 (0.15) | 미지수 (0.1~0.3) |
| 반발계수 | 고정 (0.85) | 속도 의존적 |
| 공 스핀 | 미반영 | 항상 존재 |
| 카메라 오차 | 0mm | 2~5mm |
| 테이블 수평 | 완벽 | 미세 경사 있음 |
| 도구-공 접촉 | 점 접촉 | 면적 접촉 |

> 이런 괴리를 줄이기 위해 **자동 캘리브레이션 루프**(calibration_loop.py)를 구현했습니다. 로봇이 자동으로 타격→카메라 관측→파라미터 최적화를 반복합니다.

#### 5️⃣ 2쿠션 규칙의 조합 폭발

"수구로 황구와 적구를 모두 치되, 쿠션에 2회 이상 반사" 조건은:

- 수구 타격 각도: 720개 (0.5° 단위)
- 각 각도에서 쿠션 반사 경로: **비결정적** (공 스핀, 속도에 따라 변함)
- 황구-적구 접촉 순서: 다양한 조합 가능
- 이 모든 조건을 만족하는 해: 전체 탐색 공간의 **2~5%**

단순 공식으로 이 탐색을 해결할 수 없습니다. 그래서 **물리 시뮬레이터 기반 전수 탐색** + **안전 각도 우선순위** + **2쿠션 우선 전략**을 사용합니다.

### 정리: 왜 어려운가

```
Pick & Place:    위치 → 잡기 → 놓기  (3단계, 정적, 오차 비증폭)
당구 타격:       감지 → 탐색(720각도 × 물리시뮬) → IK검증 → 궤적생성
                 → 접근 → 1ms 충격 → 관측 → 재시도  (8단계, 동적, 오차 증폭)
```

이 프로젝트의 난이도는 **"위치를 아는 것"이 아니라 "정확히 치는 것"과 "친 뒤의 결과를 예측하는 것"**에 있습니다.

- 위치를 안다고 칠 수 있는 게 아닙니다 (IK 한계, 도구 충돌)
- 칠 수 있다고 원하는 대로 가는 게 아닙니다 (물리 불확실성, 오차 증폭)
- 한 번 굴러간 공을 되돌릴 수 없습니다 (비가역적 동적 시스템)

> **프로 당구 선수가 이 과제를 어렵다고 느끼는 이유와 로봇이 어려워하는 이유는 같습니다:**
> 물리법칙은 간단해 보이지만, 2쿠션 이상에서 오차가 기하급수적으로 증폭되기 때문입니다.

## 🔬 설계 결정에 대한 기술적 근거

### Q1. "3쿠션 한다고 했다가 2쿠션으로 바꾸면 뭐가 프로젝트냐"

**답변: 3쿠션→2쿠션 우선은 물리적 현실에 기반한 최적 전략입니다.**

3쿠션을 2쿠션으로 "낮춘" 것이 아닙니다. **시스템은 3쿠션과 2쿠션을 모두 탐색**하며, 그 중 예측 정확도가 높은 것을 우선 실행합니다.

실측 데이터가 이를 증명합니다:

| 쿠션 수 | 궤적 길이 | 반사 오차 누적 | 예측 정확도 |
|---------|-----------|---------------|------------|
| 1쿠션 | ~30cm | ±0.2° | 95%+ |
| 2쿠션 | ~50cm | ±0.5° | **80~90%** |
| 3쿠션 | ~80cm | ±2~5° | **40~60%** |

3쿠션이 "더 어렵기 때문에 더 가치 있다"는 것은 **인간 기준**입니다. 로봇 시스템에서는:

- 3쿠션 궤적은 쿠션 반사 시 마찰·스핀·비선형 반발이 **3회 누적**
- 시뮬레이터와 현실의 물리 차이가 **반사마다 증폭**
- 시뮬에서 완벽한 3쿠션도 현실에서는 **50% 이상 빗나감**

반면 2쿠션은:
- 궤적이 짧아 오차 누적이 적음
- **같은 규칙(수구→쿠션2회 이상→양 타겟 접촉)**을 만족
- 실제 3쿠션 경기에서도 프로 선수들은 **최소 쿠션 경로를 우선** 선택

> 프로 당구 선수가 5쿠션 가능한 상황에서 2쿠션으로 치는 것을 "쉬운 길을 선택했다"고 하지 않습니다.
> **최소 쿠션 경로 선택은 당구의 기본 전략**이며, 이를 구현한 것은 도메인 지식의 적용입니다.

실제로 우리 시스템은 2쿠션 후보가 없으면 **자동으로 3쿠션을 시도**합니다. 3쿠션 코드를 제거한 것이 아니라, **2쿠션이 가능할 때 우선 선택**할 뿐입니다. 시뮬레이션에서 2쿠션 경로가 없을 때 3쿠션(159.5°)으로 성공한 사례가 이를 증명합니다.

### Q2. "말은 많은데 구체화된 게 없다"

**답변: 아래 항목들은 모두 코드로 구현되어 동작하는 결과물입니다.**

"구체화"의 정의가 "작동하는 시스템"이라면, 다음을 확인해 주십시오: please

| 구현 항목 | 파일 | 코드 라인 | 실측 결과 |
|-----------|------|----------|-----------|
| 720각도 전수 탐색 | `cushion_planner.py` | ~700줄 | 3368개 후보 → 2.8초 |

| ㄴ자 도구 기구학 | `trajectory_planner.py` | `plan_strike()` | 6cm×3cm 오프셋 자동 보정 |
| IK 5회 수렴 + 4단계 검증 | `ik_solver.py` | `solve_trajectory()` | 특이점/관절한계/충돌/점프 |
| PD 토크 부스트 | `robot_controller.py` | Kp 800→5000 | 0.5→1.0 m/s 타격 속도 달성 |

이 항목들은 각각 **"왜 필요한지"를 실패 데이터에서 발견하고, 코드로 해결하고, 실측으로 검증한** 결과입니다:

1. PD 게인 800에서 타격 속도 0.5 m/s → 쿠션 도달 불가 → **Kp 5000으로 부스트해서 해결**

4. 도구가 벽을 뚫는 궤적 생성 → **safe_approach_dist 동적 계산으로 해결**

> 각 항목은 "말"이 아니라 **실패→원인분석→구현→검증의 엔지니어링 사이클**을 거친 결과입니다.

### Q3. "바닥에 하드보드지 까는 건 로봇이 쉽게 하려고 환경을 바꾸는 것"

**답변: 환경 캘리브레이션은 로보틱스의 표준 절차이며, 인간에게도 동일하게 적용됩니다.**

세 가지 관점에서 반박합니다:

**① 인간과 동일 조건입니다**

이 프로젝트는 **사람과 로봇이 같은 테이블에서 같은 규칙으로** 경쟁합니다. 하드보드지를 깔면 사람도 같은 표면 위에서 칩니다. 로봇에게만 유리한 변경이 아닙니다.

**② 결함 있는 장비 보정은 당연합니다**

| 교실 당구대 현실 | 문제 |
|-----------------|------|
| 표면 거칠기 불균일 | 공이 직선으로 안 굴러감 → 물리 예측 자체가 불가능 |
| 수평 미보장 | 공이 한쪽으로 흘러감 → 정지 상태에서도 위치 변동 |
| 쿠션 탄성 불균일 | 같은 각도 같은 속도로 쳐도 반사각이 매번 다름 |

공이 **직선으로 굴러가지 않는 환경에서** 당구를 하는 것은 "어려운 도전"이 아니라 **물리적으로 불가능한 조건**입니다. 프로 당구 선수도 테이블 천이 훼손되면 교체합니다. 이것을 "환경을 바꾸는 것"이라 하지 않고 **"정상적인 경기 조건을 확보하는 것"**이라 합니다.

**③ 산업 로보틱스에서 환경 정비는 표준입니다**

- 자동차 공장: 차체 지그(jig) 정밀도 ±0.1mm로 관리 → "환경을 바꾸는 것"이 아니라 **공정 조건 확보**
- 반도체 팹: 클린룸 Class 100 유지 → 로봇이 쉽게 하려고가 아니라 **작업 가능 조건**
- 의료 로봇: 수술대 수평 보정 → **기본 셋업**

당구 테이블 표면 정비도 같은 맥락입니다. 물리 시뮬레이터가 가정하는 **"공이 직선으로 굴러감"**은 당구의 가장 기본적인 전제 조건이며, 이 전제가 깨진 환경에서는 **어떤 알고리즘도 작동할 수 없습니다**.

> 우리가 한 것은 **비정상적인 표면을 정상으로 복원**한 것이지, 난이도를 낮춘 것이 아닙니다.
> 사실 하드보드지 위에서도 마찰계수는 실제 당구대 천보다 높아서, 오히려 **공이 더 빨리 멈추고** 쿠션 도달이 더 어렵습니다.

# 당구 로봇 데모 — 왜 어려운 문제인가

> **POSTECH 로보틱스개론 팀 프로젝트**
>
> Neuromeka Indy7 6-DOF 로봇 팔 + ㄴ자형 타격 도구로 3쿠션 당구를 자율 수행하는 시스템

---

## 한 줄 요약

이 데모는 **"공 위치를 알면 치면 된다"**는 문제가 아닙니다.

로봇이 공의 정중앙을 **정확히** 맞추고, **충돌 없이** 접근한 뒤, **1ms 미만의 충격**으로 공을 보내고, **그 이후 궤적까지 예측**해야 합니다.  
수업에서 배운 IK·Manipulability·충돌 회피를 실제 환경에 적용하고, 시뮬레이션과 현실의 차이를 줄이기 위해 **반복 캘리브레이션**까지 수행한 결과물입니다.

---

## 실제 데모

### 대결 형식

| 항목 | 내용 |
|------|------|
| **대전** | 로봇 vs 사람 **1대1** |
| **승패** | **성공률이 더 높은 쪽**이 승리 |
| **시도 횟수** | 각자 **3번** (3 try) |

### 승리 조건

목표는 **3번 모두 성공**이 아닙니다.  
**사람보다 높은 성공률**을 내면 됩니다.

예: 로봇 2/3, 사람 1/3 → **로봇 승리**

### 로봇의 1샷당 프로세스

로봇은 **매 타격마다** 아래 과정을 처음부터 반복합니다. (3번 칠 때마다 3회 수행)

```
비전 인식 → PyBullet 시뮬 타격 계획 → IK·충돌 검증 → 실로봇 궤적 실행
```

1. **비전 (RealSense)** — 테이블 위 공 위치 검출  
2. **시뮬레이션** — 동일 배치로 PyBullet에서 각도·속도 탐색 및 궤적 생성  
3. **실행** — 검증된 궤적을 실제 로봇에 전달하여 타격  

사람은 같은 테이블·같은 규칙으로 직접 칩니다.  
로봇만 유리한 환경이 아니라, **동일 조건에서 성공률을 겨루는 게임**입니다.

## 1. Pick-and-Place가 아닌 이유

| | Pick & Place | 당구 타격 |
|---|---|---|
| **제어 대상** | 위치 (x, y, z) | 위치 + 속도 + 방향 + 타이밍 |
| **동작 속도** | 느림 (~0.1 m/s) | 빠름 (**1.0+ m/s**) |
| **접촉** | 정적 파지 (grasp) | **충격 (impulse, <1ms)** |
| **오차 전파** | 거의 없음 | **기하급수적 증폭** |
| **되돌리기** | 즉시 재배치 가능 | **불가능** (공은 이미 굴러감) |

Pick-and-place에서 1mm 오차는 1mm입니다.  
당구에서 **0.5° 각도 오차**는 2쿠션 이후 **수십 cm** 궤적 차이로 커집니다.

## 2. 로보틱스개론 수업 내용의 실제 적용

### 2.1 역기구학 (Inverse Kinematics)

로봇 팔은 당구 큐가 아닙니다. 6개 관절의 회전 조합으로 **직선 타격 궤적**을 만들어야 합니다.

- **Damped Least Squares (DLS) IK** — 5회 반복 수렴으로 곡선 궤적 방지
- **관절 한계 검증** — Indy7 URDF 기준 관절 범위 초과 여부 확인
- **궤적 사전 검증** — 실행 전 전체 경로에 대해 IK 해 존재 여부 확인

```
Home → 상공(25cm) → Ready(공 뒤 8~10cm) → Impact(공 중심) → Follow-through → Home
```

### 2.2 Manipulability를 고려한 제어

특이점(singularity) 근처에서는 관절 속도가 발산하고, 타격 순간의 미세 오차가 결과를 바꿉니다.

- Manipulability index \( w = \sqrt{\det(J J^T)} \) 계산
- **Manipulability ≥ 0.015** 미만인 자세는 후보에서 제외 (글랜싱 블로우 방지)
- 적응형 DLS damping — 특이점 근방에서 감쇠 계수 자동 증가

### 2.3 Collision Avoidance (충돌 회피)

당구대 세팅에서는 두 가지 제약이 동시에 걸립니다.

| 제약 | 의미 |
|------|------|
| **공 비접촉** | 접근·정렬 단계에서 로봇/도구가 공을 건드리면 안 됨 |
| **벽 비충돌** | 로봇 메쉬·ㄴ자 도구가 쿠션/벽에 닿으면 안 됨 |

이 조건을 만족하는 관절 각도와 경로만 남깁니다.

- 궤적 전체 10% 샘플링 IK 검증
- 바디 뚫림 검사 (z < 0.30m)
- 관절 점프 검사 (< 0.3 rad)
- **`safe_approach_dist`** — 벽 근처 공일수록 접근 거리를 동적으로 줄여 도구가 벽을 뚫지 않도록 계산

실측 기준, 전체 후보 중 **약 15~25%만** IK + 충돌 + 벽 검사를 통과합니다.

## 3. 현재 파이프라인 (요약)

```
[현실] ──비전──► 공 좌표 (cue, tgt1, tgt2)
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
 Headless        GUI PyBullet     실로봇
 (각도·속도 탐색)  (IK·로봇 궤적)   (관절 재생)
```

1. **비전 (RealSense)**: 테이블 위 3공 위치 검출 → PyBullet 좌표 변환
2. **Headless 탐색**: 최적 타격 각도·속도·공 궤적 예측 (로봇 없음)
3. **GUI 시뮬**: IK로 로봇 관절 궤적 생성·검증 후 시뮬에서 타격
4. **실로봇**: GUI에서 저장한 관절 궤적을 MoveJ로 재생

> Headless에서 나오는 것은 **로봇 관절 경로가 아니라** “어느 각도·속도로 칠지”입니다.  
> 상세 과정은 **§10 전체 시스템 동작 과정** 참고.

## 4. Sim-to-Real Gap — 우리가 모르는 것들

시뮬레이터는 이상적인 물리 법칙을 가정합니다. 현실은 그렇지 않습니다.

| 파라미터 | 시뮬 (초기값) | 현실 |
|----------|--------------|------|
| 바닥 마찰계수 | 고정 (0.15) | **미지수** — 재질·온도·습도에 따라 변함 |
| 쿠션 반발계수 | 고정 (0.8) | **측정 불가** — 비선형, 속도·입사각 의존 |
| 공 스핀 | 미반영 | 항상 존재 |
| 카메라 오차 | 0 mm | 2~5 mm |
| 테이블 수평 | 완벽 | 미세 경사 존재 |

**쿠션 반발계수는 사전에 정확히 알 방법이 없습니다.**  
교과서의 "입사각 = 반사각" 공식은 1쿠션까지 근사적으로 맞지만, 2쿠션 이상에서는 오차가 누적됩니다.

## 5. 테이블 표면 재질 — 왜 하드보드지를 깔았는가

### 5.1 기존 당구대 재질의 문제

교실 당구대 원판 재질은 마찰계수가 너무 커서, 데모에 필요한 **기본 물리 조건**부터 충족되지 않았습니다.

| 기존 재질 문제 | 데모/알고리즘에 미치는 영향 |
|----------------|---------------------------|
| **마찰 과대** | 공이 쿠션에 닿기 전에 멈춤 → 2쿠션/3쿠션 자체 불가 |
| **표면 불균일** | 직선 이동 불가 → 궤적 예측·캘리브레이션 모두 무의미 |
| **방향별 마찰 차이** (결·주름) | 시뮬은 등방성 가정 → 샷마다 결과가 달라짐 |
| **습도·먼지 민감** | 데모 당일 조건이 매번 달라짐 |

이것은 "어려운 도전"이 아니라 **물리적으로 작업 조건이 깨진 상태**입니다.  
Pick-and-place에서 테이블이 기울어져 있으면 알고리즘과 무관하게 실패하는 것과 같습니다.

### 5.2 기존 재질 vs 하드보드 세팅

| 기준 | 기존 재질 | 하드보드 + 표준 상판 |
|------|-----------|----------------------|
| 데모 성공률 | 낮고 불안정 | 높고 재현 가능 |
| 캘리브레이션 | 샷마다 drift | 수렴 가능 |
| 발표 스토리 | "환경이 너무 나빠서…" | "비정형 환경 → 표준화 → 캘리브레이션" |
| 난이도의 본질 | **불확실성**이 높음 | **제어·물리 모델링** 난이도는 그대로 |

**결론:** 데모 목적이라면 기존 재질을 고집하기보다, **균일하고 캘리브레이션 가능한 표면**을 확보하는 것이 맞습니다.  
산업 로봇에서 지그·작업면을 맞추는 것과 같은 **표준 공정 조건 확보**입니다.  
사람과 로봇이 **같은 표면** 위에서 치므로, 로봇에게만 유리한 변경이 아닙니다.

### 5.3 재질 선택 가이드

목표는 시뮬 기본값(`lateral=0.15`, `rolling=0.012`)과 비슷한 범위입니다.

- **횡 마찰 (lateral):** 0.10 ~ 0.20
- **구름 마찰 (rolling):** 0.008 ~ 0.015
- **1 m/s 타격 후** 공이 **최소 20~30 cm 이상** 굴러가야 쿠션 도달 가능

| 순위 | 재질 | 특성 |
|------|------|------|
| **1순위** | 하드보드(MDF) + **스피드 천** (Speed cloth) | 마찰 특성이 비교적 알려져 있고, 직진성·미끄러짐 균형 우수. 데모용으로 당구 느낌 유지 |
| **2순위** | 하드보드 + **얇은 PET/아크릴** (0.5~1 mm) | 마찰 매우 낮음 → 공이 잘 굴러감. 단, 너무 미끄러우면 미세 각도 오차도 커짐 |
| **3순위** | 하드보드 + **무광 라미네이트/색지** | 예산·시간이 빠듯할 때 가능. 광택지·일반 종이·골판지는 피할 것 (마찰 불균일, 주름, 습기 흡수) |

**비추천:** 기존 거친/오염된 원판 그대로, 골판지, 광택이 강한 비닐

### 5.4 하드보드 세팅 체크리스트

표면 재질만큼 **설치 품질**이 중요합니다.

1. **MDF/하드보드**로 원판 요철·틈 먼저 메우기
2. **접착** — 양면테이프보다 풀/스프레이 접착 + 롤러로 기포·주름 제거
3. **쿠션 높이와 수평** — 공 중심과 쿠션 립 정렬 (±1 mm 수준)
4. **테두리 고정** — 타격 시 표면이 밀리지 않도록
5. **데모 전날 동일 조건 테스트** — 같은 위치에서 5샷, 이동 거리·편차 기록

핵심은 "최저 마찰"이 아니라 **"낮고 + 균일하고 + 캘리브레이션 가능"**한 재질을 고르는 것입니다.

## 6. 캘리브레이션 — 현실과 시뮬의 차이를 줄이는 노력

`calibration_loop.py`로 **좌표 → 물리(2a → 2b)** 순서로 캘리브레이션합니다.

### Phase 1: 좌표 캘리브레이션

- 카메라로 공 검출 → 로봇이 해당 위치로 접근
- 실제 접촉 여부로 **카메라↔시뮬 좌표 오프셋** 자동 보정
- 출력: `calibration_position_offset.json`

### Phase 2: 물리 파라미터 — **직선 / 1쿠션 분리** (권장)

**핵심:** 쿠션·각도 탐색 없이, **실측 정지 위치 vs 시뮬 정지 위치**만 비교합니다.

| 단계 | 실험 | 튜닝 파라미터 | 쿠션 |
|------|------|---------------|------|
| **2a 직선** | 고정 각도(0°) 직진 샷 | `lateral_friction`, `rolling_friction`, `speed_gain_scale` | 불필요 |
| **2b 1쿠션** | 쿠션에 1회 반사 | `cushion_restitution` | **필요** |

- 직선 샷만으로는 **쿠션 반발계수를 맞출 수 없음**
- `plan_shot()` / 720각도 탐색은 캘리브에 **넣지 않음** (물리 loss 오염 방지)

### 기존 코드 (현재 `calibration_loop.py` Phase 2) — 참고

| 단계 | 동작 |
|------|------|
| 1 | `plan_shot()`으로 각도·속도 **자동 선택** (2/3쿠션 후보 가능) |
| 2 | 실로봇 타격 → `wait_real_balls_stop()`으로 **3공 최종 XY** 기록 |
| 3 | `simulate_strike()` — Headless에서 동일 조건, 공에 속도 직접 부여 |
| 4 | `Loss = Σ ‖실측 final − 시뮬 final‖²` (수구·황구·적구) |
| 5 | **Nelder-Mead**로 6개 파라미터 **동시** 조정 |

→ 마찰·반발을 **따로 측정**하는 게 아니라, “최종 위치가 비슷해지는” 6개 PyBullet 계수를 **블랙박스 피팅**.

**한계·개선안·최적 튜닝 방법:** **§13** 전체 참고.

## 7. IK 이후 — 타격과 사후 예측

IK로 "칠 수 있는 경로"를 찾는 것에서 끝나지 않습니다.

### 7.1 타격 실행

- **PD 토크 제어** (Kp=5000, Kd=200) — 1.0 m/s 수평 타격 속도 달성
- PD gain 튜닝을 통해 **최적 타격 속도** 탐색 (기존 Kp=800에서는 0.5 m/s로 쿠션 도달 불가)
- ㄴ자 도구 오프셋 (6cm 수직 + 3cm 수평) 자동 보정

### 7.1.1 PD Gain 튜닝 — 왜 필요한가

PD gain은 **로봇 팔이 목표 궤적을 얼마나 빠르고 정확히 따라가느냐**를 결정합니다.  
당구 데모에서는 타격 순간의 EE(엔드이펙터) 속도가 공에 얼마나 전달되느냐와 직결됩니다.

#### PD 제어 구조

Indy7은 **관절 토크 제어 (computed torque + PD)** 로 동작합니다.

```
τ = M(q) · [ q̈_des + Kp·(q_des − q) + Kd·(q̇_des − q̇) ] + C(q,q̇)·q̇ + g(q)
```

| 항 | 역할 |
|---|---|
| **Kp (비례 gain)** | 목표 관절각과 현재 관절각 차이를 줄임 → **위치 추종 강도** |
| **Kd (미분 gain)** | 목표 속도와 현재 속도 차이를 줄임 → **진동·오버슈트 억제** |
| **M, C, g** | 관절 동역학 보상 (computed torque) |

Kp가 크면 목표에 빨리 붙지만 **오버슈트·진동**이 커지고, Kd가 크면 부드럽지만 **반응이 느려집니다**.

#### Pick-and-Place와의 차이

Pick-and-place는 느리게 움직이므로 기본 gain으로도 충분합니다. 당구는 다릅니다.

- 타격 구간 속도 목표: **~1.0 m/s**
- 접촉 시간: **1 ms 미만**
- 1~2 mm 위치 오차 → 글랜싱 블로우 또는 각도 오차

**"느리게 정확히 가는 것"**과 **"빠르게 정확히 치는 것"**은 서로 다른 gain이 필요합니다.

#### 2단계 Gain 전략

코드에서는 **접근(Approach)** 과 **타격(Strike)** 에 gain을 나눠 적용합니다.

| 단계 | Kp | Kd | 목적 |
|------|-----|-----|------|
| **Approach** (공 앞 Ready까지) | 800 | 40 | 부드럽게 접근, 테이블·공 비접촉 유지 |
| **Strike** (Ready → Impact) | **5000** | **200** | 짧은 구간에서 **최대 속도·직선성** 확보 |

기본 PyBullet 프레임워크 값은 `Kp=500, Kd=20`인데, 이 값으로는 타격 속도가 **~0.5 m/s** 수준에 머물러 **쿠션까지 공이 도달하지 못했습니다**.

`run_maze.py`에서 strike 구간 gain을 올려 실측 **0.5 → 1.0+ m/s** 타격 속도를 달성했습니다.

#### 튜닝 과정 (실측 기반)

1. **Kp=800 (기본 boost)** — 접근은 안정적이나 strike 구간 속도 부족 → 공이 쿠션 전에 멈춤
2. **Kp=5000, Kd=200 (strike boost)** — 240 Hz로 궤적 waypoint 스트리밍, EE가 strike 직선 궤적을 더 정확히 추종
3. **Kd를 함께 올린 이유** — Kp만 올리면 관절 진동·오버슈트 → 타격 직전 EE 흔들림. Kd=200으로 감쇠를 키워 **충격 직전 안정성** 확보
4. **Headless 시뮬과 GUI 시뮬 동일 gain** — `cushion_planner.py`의 PD 검증도 Kp=5000, Kd=200 사용 → 시뮬 탐색 결과와 실제 실행 물리 일치

#### PD Gain과 공 속도의 관계

PD 튜닝만으로 끝나지 않습니다. 도구→공 속도 전달에는 별도 보정이 있습니다.

```
공 속도 ≈ 도구(EE) 속도 × BALL_SPEED_GAIN × speed_gain_scale
```

- `BALL_SPEED_GAIN`: 1D 탄성 충돌 모델 (도구 질량, 반발계수 기반)
- `speed_gain_scale`: **캘리브레이션**으로 실측 보정 (`calibration_loop.py` Phase 2)

PD gain은 **로봇이 목표 속도로 움직이게** 하고,  
`speed_gain_scale`은 **그 속도가 공에 얼마나 전달되는지** 맞춥니다.

> **Approach는 낮은 gain으로 안전하게 접근하고, Strike는 높은 gain으로 1 m/s 직선 타격을 만든다.**  
> PD 튜닝 없이는 공이 쿠션에 닿기 전에 멈추므로, 당구 알고리즘 자체가 의미를 잃는다.

### 7.2 타격 이후 상황 예측

- PyBullet 헤드리스에서 3공 + 쿠션 물리 시뮬레이션
- 접촉 이벤트 추적: 수구 → 쿠션 반사 → 황구/적구 접촉
- **2쿠션 우선 전략** — 궤적이 짧을수록 예측 정확도 ↑ (3쿠션은 오차 3회 누적)

### 7.3 각도 탐색 알고리즘

720각도(0.5° 단위) × 3속도 = **2,160개 후보** 전수 탐색 후 필터링:

| 우선순위 | 각도 범위 | 특성 |
|----------|-----------|------|
| 0 (최우선) | 50° ~ 180° | 안전 — 글랜싱 블로우 없음 |
| 1 | 30°~50°, 180°~330° | 보통 |
| 2 (제외) | 0°~30°, 330°~360° | 위험 — 공을 빗겨침 |

**벽에 붙은 공**은 로봇 작업공간·도구 충돌 제약으로 접근 자체가 어렵습니다.  
이 구간을 제외하고, **공이 테이블 중앙부에 있을 때 성공률 70% 이상**을 유지하도록 각도·속도 후보를 선별합니다.

## 10. 전체 시스템 동작 과정 (상세)

### 10.1 세 계층: Headless → GUI → 실로봇

헤드리스 PyBullet 결과를 **그대로** 실로봇에 넣는 구조가 **아닙니다**.  
각 단계에서 전달되는 데이터가 다릅니다.

| 단계 | 역할 | 입력 | 출력 |
|------|------|------|------|
| **① Headless** | “어디로·얼마나 세게 칠지” 탐색 | 공 3개 좌표 | `angle`, `speed`, `score`, `cue_path`(2D 예측) |
| **② GUI 시뮬** | 로봇이 **어떻게 움직일지** 생성 | 최적 각도·속도 | SE3 궤적 → **관절 궤적** `q_traj` |
| **③ 실로봇** | 검증된 동작 재현 | GUI에서 저장한 `q_traj` | MoveJ 타격 |

```
[Headless PyBullet]  CushionShotPlanner._fast_ball_search()
   공에 속도 직접 부여, 로봇·도구 없음
        │  angle, strike_dir, score, ball_path
        ▼
[GUI PyBullet]  StrikeTrajectoryPlanner + IKSolver
   Approach → Strike → Follow-through
        │  q_traj (관절각 시퀀스)
        ▼
[실로봇 Indy7]  7TestCode.replay_trajectory_on_real()
   MoveJ로 저장 궤적 재생
```

**핵심 파일:** `cushion_planner.py` (탐색) → `trajectory_planner.py` + `ik_solver.py` (궤적) → `7TestCode.py` (실기)

### 10.3 각도 탐색 알고리즘 (`CushionShotPlanner.plan_shot`)

구현: `project/physics/cushion_planner.py`

#### Stage 1: `_fast_ball_search()` — Grid + Annealing

매 `plan_shot()` 호출 시 **새 Headless 월드** (`p.DIRECT`) 생성.

| 항목 | 값 |
|------|-----|
| 각도 | **720개** (0~360°, 0.5° 간격) |
| 속도 | **3단계** (최대 ~1.85 m/s, 75%, 50%) |
| 초기 시뮬 횟수 | **2,160회** (720 × 3) |
| Annealing | 상위 10% 주변 **2라운드** 정밀화 (`ANNEAL_SIGMA_ANGLE`: 30°→10°→3°) |

각 시뮬 (`_simulate_ball_only`):

1. 3공 위치 리셋  
2. 수구에 `(vx, vy) = speed × (cos θ, sin θ)` 부여  
3. PyBullet 240Hz 구름, 접촉 이벤트 기록 (`c`=쿠션, `t1`, `t2`)  
4. `_score_result()`로 점수 산출  

**점수 체계:**

| 조건 | 점수 |
|------|------|
| 쿠션 ≥2 + t1·t2 히트 | **+3000** (2쿠션) |
| 쿠션 ≥3 + t1·t2 히트 | **+3000** (3쿠션) |
| t1/t2만 | +500 each |
| 테이블 이탈 | −2000 |

#### Stage 2: 후보 필터 (`_filter_reachable_diverse`)

| 필터 | 내용 |
|------|------|
| 각도 우선순위 | 50°~180° 최우선 (글랜싱 방지) |
| 도달 가능 | Ready 위치가 로봇 반경 0.65 m 이내 |
| 벽 안전 | `safe_approach_dist` — 큐팁이 쿠션에 닿지 않도록 접근 거리 축소 |
| 각도 다양성 | 이미 선택된 후보와 5° 이내 중복 제외 |
| 최종 정렬 | **쿠션 수 적은 순** → score 높은 순 (2쿠션 > 3쿠션) |

**참고:** 로봇 PD Headless 검증(`_create_robot_env`) 코드는 있으나, **현재는 생략**하고 fast search 결과를 GUI에 직접 전달합니다. (GUI와 headless 로봇 물리 차이로 인한 오차 방지)

### 10.6 자율 상태 머신 (`state_machine.py`)

```
  ↑                      │
```

| 상태 | 동작 |
|------|------|

| **OBSERVE** | 쿠션 횟수·타겟 히트로 성공 판정 |

Manipulability ≥ 0.015, 관절 한계, 바디 뚫림, `safe_approach_dist` 검사 포함.

### 11.2 단계 A — 환경 (표면·수평)

**작업**

1. MDF 평탄화 + 상판 접착 (주름·기포 제거)

### 11.4 단계 C — 큐팁 오프셋 보정 (EE ≠ 타격점)

ㄴ자 도구는 EE에서 **수평 30 mm + 수직 60 mm** 떨어진 곳이 실제 큐팁입니다.

5. 4점의 기하학적 중심 = 실제 타격점 추정

### 11.6 샷 직전 미세 정렬 루프 (ALIGN)

**삽입 위치:** `7TestCode.py` / `state_machine._strike()` 에서  
`plan_strike` **이후**, `execute_trajectory` **직전**.

```
SCAN → THINK → [ALIGN] → STRIKE → OBSERVE
```

#### ALIGN 상태 머신

```
ALIGN_IDLE
  → detect_balls()  (cue_xy)
  → FK로 큐팁 XY 예측: tip_xy = f(EE, TOOL_HORIZONTAL_EXT, TOOL_YAW_OFFSET, strike_dir)
  → err = cue_xy - tip_xy
  → |err| < 1.5mm ? STRIKE : ALIGN_MOVE
ALIGN_MOVE
  → strike_dir에 수직인 평면에서 EE를 err의 50% 만큼 이동 (1 step)
  → max 5회 반복
  → 초과 시 WARNING + 그대로 STRIKE (또는 샷 스킵)
```

#### 파라미터 (`config.py` 추가 권장)

```python
ALIGN_ENABLE = True
ALIGN_MAX_ITER = 5
ALIGN_TOLERANCE_MM = 1.5      # 큐팁-공 중심 XY
ALIGN_STEP_RATIO = 0.5          # err의 50%씩 수렴
ALIGN_MAX_SHIFT_MM = 3.0        # 1 step 최대 이동
ALIGN_STRIKE_SPEED = 0.0        # 정렬 중에는 이동만 (strike 없음)
```

#### 구현 스케치 (`project/align_servo.py` 신규)

```python
def align_tip_to_ball(controller, ball_pos, strike_dir_3d, ik, traj_planner):
    for i in range(ALIGN_MAX_ITER):
        cue_xy = detect_balls()[0][:2]  # 또는 마지막 SCAN 재사용
        T_ee = controller.get_current_T()
        tip_xy = predict_tip_xy(T_ee, strike_dir_3d)  # ㄴ자 기구학
        err = np.array(cue_xy[:2]) - tip_xy
        if np.linalg.norm(err) * 1000 < ALIGN_TOLERANCE_MM:
            return True, err
        # err 방향으로 EE만 소폭 이동 (strike_dir 수직 성분)
        delta = _lateral_delta_SE3(err, strike_dir_3d, ALIGN_STEP_RATIO)
        T_next = T_ee + delta
        q = ik.solve_step(controller.get_current_q(), T_next)
        controller.movej(np.degrees(q.flatten()), ...)  # 느린 movej
    return False, err
```

**주의**

- ALIGN 중 **공과 접촉 금지** — `approach_dist` 유지, 속도 ≤ 0.1 m/s  
- 비전 노이즈 때문에 **5회 median** 또는 **2프레임 연속** 통과 시만 OK

## 13. 캘리브레이션 코드 수정 가이드 (`calibration_loop.py`)

기존 Phase 2의 **한계**, **수정 포인트**, **권장 파라미터 튜닝 방법**을 정리합니다.

### 13.0 기존 코드가 파라미터를 찾는 방식

**파일:** `calibration_loop.py` — `run_physics_calibration()` → `simulate_strike()` → `evaluate()` → `optimize_parameters()`

**최적화 대상 6개 (`PARAM_NAMES`):**

| 파라미터 | PyBullet 적용 |
|----------|----------------|
| `lateral_friction` | 테이블·공 횡 마찰 |
| `rolling_friction` | 공 구름 마찰 |
| `ball_restitution` | 공 반발 |
| `cushion_restitution` | 쿠션 4면 반발 |
| `speed_gain_scale` | `ball_speed = strike_speed × gain × BALL_SPEED_GAIN` |
| `table_y_offset` | 테이블 중심 Y 보정 |

**Loss (현재):**

```python
# evaluate(params, trials) — 요약
for trial in trials:
    sim = simulate_strike(trial, params)  # headless, 3공, 쿠션 포함
    for key in ['cue_final', 'target_final', 'ball2_final']:
        loss += sum((sim[key] - trial[key])**2)
# minimize(loss) — Nelder-Mead, 6차원
```

**의도:** 시뮬 최종 위치 ≈ 실측 최종 위치  
**문제:** 어떤 물리량을 맞추는지 **식별(identifiable)하기 어렵다.**

---

### 13.1 기존 캘리브레이션의 한계

#### A. 실험 설계

| 한계 | 설명 |
|------|------|
| **`plan_shot()` 의존** | trial마다 각도·쿠션 수가 달라 **조건이 섞임** |
| **3공 loss** | 황구·적구 final은 직선/마찰 캘리브에 **노이즈** |
| **최종 위치만 사용** | 초기 속도·굴림 거리·반사 각도 미사용 → 마찰/속도 **분리 어려움** |
| **trial ~5회** | 조건이 제각각이면 **통계 부족** |

#### B. 물리·식별성

| 한계 | 설명 |
|------|------|
| **마찰 ↔ 속도 스케일** | 둘 다 굴림 거리를 줄임 → **다중해** (같은 loss를 여러 조합으로 맞출 수 있음) |
| **쿠션 반발 ↔ 공 반발** | 3공 충돌·쿠션 반사가 섞이면 구분 불가 |
| **`table_y_offset`** | Phase 1 좌표 오프셋과 **중복** 가능 |
| **중심 미타격** | 좌표·큐팁 오차가 **잘못된 마찰/반발**로 흡수됨 |

#### C. 최적화 알고리즘

| 한계 | 설명 |
|------|------|
| **Nelder-Mead 6D** | 파라미터 **상관** → 지역 최솟값, 해석 어려움 |
| **직선 위주 trial** | `cushion_restitution`은 loss에 **거의 민감하지 않음** |
| **시뮬 ≠ 실기 타격** | 시뮬은 공에 속도 직접 부여, 실기는 PD → `speed_gain_scale`이 **전부 흡수** |

**결론:** 6개 동시 Nelder-Mead **비추천**. **2a(직선) → 2b(1쿠션)** 분리 + **순차 1D grid** 권장.

### 13.2 수정 우선순위 (체크리스트)

| 우선순위 | 수정 내용 | 대상 |

### 13.4 Phase 2a — 직선 샷 물리 캘리브 (구체화)

#### 실험 프로토콜

| 항목 | 값 |
|------|-----|
| 수구 위치 | 테이블 중앙 근처 `(CX, CY - 0.10, ball_h)` |
| 황구·적구 | **배치 안 함** 또는 시뮬에서만 (loss에 미사용) |
| 타격 방향 | **+X 고정** (`strike_angle = 0`) |
| 속도 | `[0.6, 0.8, 1.0] m/s` × 각 2샷 = **6 trials** |
| `plan_shot` | **호출하지 않음** |

#### Trial 데이터 구조 (변경)

```python
trial_linear = {
    'type': 'linear',
    'cue_start': np.array([x, y]),
    'strike_angle': 0.0,
    'strike_speed': 0.8,           # EE/도구 명령 속도
    'cue_final': np.array([x, y]),   # 실측 정지 위치 (vision)
    'roll_distance': float,        # ||cue_final - cue_start||
    # 선택: 타격 직후 100ms 속도 (카메라 연속 or sim diag)
    'cue_v0_meas': None,
}
```

#### 시뮬 (`simulate_linear_strike`)

- `_fast_ball_search`와 동일: headless, 공에 `(vx,vy)` 직접 부여
- **쿠션 mesh 생략 가능** (직선만 보면 됨) → 빌드 단순화
- `ball_speed = strike_speed * speed_gain_scale * BALL_SPEED_GAIN` (기존식 유지)

#### Loss (`evaluate_linear`)

```python
def evaluate_linear(params, trials):
    # params: [rolling_friction, lateral_friction, speed_gain_scale]
    loss = 0.0
    for t in trials:
        sim = simulate_linear_strike(t, params)
        # 1) 정지 위치 (주 loss)
        loss += np.sum((sim['cue_final'] - t['cue_final'])**2)
        # 2) 굴림 거리 (보조)
        sim_roll = np.linalg.norm(sim['cue_final'] - t['cue_start'])
        meas_roll = t['roll_distance']
        loss += (sim_roll - meas_roll)**2
        # 3) 선택: 초기 속도
        if t.get('cue_v0_meas') is not None:
            loss += 0.5 * (sim['cue_v0'] - t['cue_v0_meas'])**2
    return loss
```

**최적화 파라미터 (2a만)**

| 파라미터 | bounds |
|----------|--------|
| `lateral_friction` | 0.05 ~ 0.5 |
| `rolling_friction` | 0.001 ~ 0.05 |
| `speed_gain_scale` | 0.7 ~ 1.5 |

`ball_restitution`, `cushion_restitution`, `table_y_offset` → **2a에서 고정** (기본값 또는 Phase 1 이후 고정)

#### 실험 프로토콜

| 항목 | 값 |
|------|-----|
| 수구 위치 | **아래쪽 쿠션에서 15~20 cm** `(CX, CY - W/4)` |

```python
}
```

#### 시뮬 (`simulate_one_cushion_strike`)

- headless에 **쿠션 4면 필수**
- 2a에서 고정된 `lateral_friction`, `rolling_friction`, `speed_gain_scale` 사용
- **최적화 변수:** `cushion_restitution` (1개, 또는 + `ball_restitution`)

#### Loss (`evaluate_cushion`)

```python
def evaluate_cushion(params, trials, fixed_physics):
  # params: [cushion_restitution] 또는 [cushion_restitution, ball_restitution]
  # fixed_physics: 2a 결과에서 로드
    loss = 0.0
    for t in trials:
        sim = simulate_one_cushion_strike(t, params, fixed_physics)
        loss += np.sum((sim['cue_final'] - t['cue_final'])**2)
    return loss
```

**통과:** 3 trial 평균 위치 오차 < 5 cm.

### 13.10 권장 파라미터 튜닝 방법 (최적안)

원칙: **식별 가능한 것만**, **순서대로**, **이전 단계는 freeze**, **단순한 loss**.

#### 계층적 캘리브레이션 순서

```
[0] 표면·수평 (하드보드+상판)
[1] Phase 1: 좌표 (calibration_position_offset.json)
[2] (권장) 큐팁 4방향 접촉 오프셋
[3] Phase 2a: 직선 — lateral, rolling, speed_gain_scale
[4] Phase 2b: 1쿠션 — cushion_restitution
[5] 데모: plan_shot + ALIGN
```

#### Phase 2a — 순차 1D grid (Nelder-Mead 6D 대신)

**Step A** — `speed_gain_scale = 1.0` 고정

1. `lateral_friction` grid: 0.08 ~ 0.25 (10점)  
2. `rolling_friction` grid: 0.005 ~ 0.02 (8점)  
3. 조합 중 **굴림 거리 RMSE** 최소 선택  

**Step B** — Step A 마찰 **고정**

1. `speed_gain_scale` grid: 0.8 ~ 1.3 (10점)  
2. (선택) 초기 속도 `|v0|` 맞춤  

**통과 기준:** `mean(sim_roll / meas_roll) ∈ [0.85, 1.15]`

#### Phase 2b — 1D grid on `cushion_restitution`

- 2a 결과 **고정**  
- `cushion_restitution` grid: 0.5 ~ 0.95 (20점)  
- Loss: 수구 `cue_final` XY 오차만  

**통과:** 3 trial 평균 위치 오차 **< 5 cm**

#### Loss 설계 (권장)

**2a (수구만):**

```text
L_2a = 1.0 · Σ ‖p_final^sim − p_final^meas‖²
     + 2.0 · Σ (d_roll^sim − d_roll^meas)²
     + 1.0 · Σ (|v0^sim| − |v0^meas|)²    # v0 측정 시만
```

**2b:**

```text
L_2b = Σ ‖p_final^sim − p_final^meas‖²
```

#### 최적화 알고리즘 비교

| 방법 | 2a (3파라) | 2b (1파라) | 비고 |
|------|------------|------------|------|
| **순차 1D grid** | ★★★ | ★★★ | 구현·해석 쉬움, **권장** |
| Golden section | ★★ | ★★★ | 1D에 적합 |
| Nelder-Mead 3D | ★★ | — | 2a 대안 |
| **Nelder-Mead 6D (기존)** | ★ | — | **폐기 권장** |

trial 수가 6~10개 수준이면 **grid + 그래프 확인**이 NM 6D보다 안정적입니다.

## 14. 정리

> **프로 당구 선수가 이 과제를 어렵다고 느끼는 이유와 로봇이 어려워하는 이유는 같습니다.**
> 물리법칙은 간단해 보이지만, 2쿠션 이상에서 오차가 기하급수적으로 증폭되기 때문입니다.

- 위치를 **안다고** 칠 수 있는 게 아닙니다 → IK 한계, 도구 충돌, 벽 제약
- 칠 **수 있다고** 원하는 대로 가는 게 아닙니다 → 마찰·반발·스핀 불확실성
- 한 번 굴러간 공을 **되돌릴 수 없습니다** → 비가역적 동적 시스템

이 데모는 로보틱스개론 수업의 IK·Manipulability·충돌 회피 이론을 **실제 물리 환경**에 적용하고,  
시뮬레이션과 현실의 차이를 **좌표(Phase 1) → 직선 물리(2a) → 1쿠션 반발(2b)** 순의 캘리브레이션으로 줄이려는 엔지니어링 노력의 결과입니다.
